Case Studies

The following cases are selected based on the core requirements of the Grab AI Platform Architect role — "Turn AI Into Engineering Leverage" and "Build Reliability That Feels Invisible" — combined with my own understanding of the position. I believe the key to this role is not just technical architecture capability, but the ability to transform AI from a tool into engineering leverage, and to evolve reliability from firefighting into a systematic discipline. The three cases demonstrate: building an AI engineering system from zero, establishing observability for rapid fault detection, and productizing troubleshooting experience into AI-driven automation.

Case 1: Building an AI Engineering System for a Team with No AI Experience

Mapped to: AI-Assisted Engineering / 0→1 Tech Stack

Context: The company's existing tech stack was based on Java microservices — complex to deploy and unfriendly to AI tooling. We needed to build a tech stack for a brand-new AI-powered audit product, with no prior AI product development experience on the team.

Challenges: ① How to select an architecture that supports rapid AI product iteration while enabling non-developer backgrounds to contribute effectively? ② How to make AI tools (Cursor, etc.) genuinely productive in daily development, not just novelties?

Decisions & Actions:

  • Selected TypeScript + Next.js monolith over Java microservices. Rationale: end-to-end type safety reduces interface errors; a single codebase gives AI tools full context; deployment complexity drops from N services to 1.
  • Established three engineering principles — end-to-end type safety, no implicit dependencies, minimal infrastructure dependencies — each directly serving the goal of "enabling AI to write correct code."
  • Developed custom skills (dev-standard-compliance, code-review, quality-report) that run automatically in local and CI environments. Continuously refined prompts and project structure to help AI focus on key issues and reduce noise.

Results: First version launched in 2 months; first enterprise customer delivered in 5 months. The team adopted an "everyone is an engineer" model where PMs and architects write code directly.

Case 2: Achieving Minute-Level Fault Detection Through Observability

Mapped to: Reliability Engineering / Detection & Recovery

Context: After the AI audit product went live, we needed to build a production observability system to ensure issues could be detected and diagnosed quickly.

Design Approach: Integrated OpenTelemetry to collect Traces and Metrics, connected to the Guance Cloud platform, using PromQL for querying. Focused on three signal layers: request-level trace chains, system-level resource metrics (CPU/memory/network), and K8s-level event streams.

Field Validation: In the second week after launch, monitoring detected a pod restart loop. Investigation: ① Stable RPS ruled out traffic spikes; ② CPU usage spiked while memory stayed within limits, pointing to a CPU-bound issue rather than memory leak; ③ K8s events confirmed health check timeouts triggering restarts; ④ Root cause: a CPU-intensive task blocking the Node.js main thread, preventing HTTP health check responses.

Value: Time from anomaly detection to root cause identification: minutes. Subsequently migrated such tasks to Worker threads and added CPU usage alerting rules, establishing a reusable troubleshooting pattern.

Case 3: From Troubleshooting to AI-Automated Analysis — Closing the Loop

Mapped to: AI-Assisted Engineering / Agentic Operational Automation

Context: After identifying the Node.js main thread blocking issue in Case 2, I realized this class of problems would recur — any CPU-intensive operation could block the event loop, but existing monitoring could only show "high CPU" as a symptom, not pinpoint which code caused the blocking.

Phased Approach:

  • Enhanced observability: Added event loop delay metrics and enabled CPU profiling data collection. This enabled not just detecting "the main thread is stuck" but seeing "where it's stuck and for how long."
  • Fixed typical scenarios: Discovered and fixed multiple blocking scenarios based on the new metrics, accumulating troubleshooting patterns from metric anomalies to code-level root causes.
  • Productized troubleshooting into an AI skill: Built a Claude Code skill that enables natural language-triggered analysis — input "analyze yesterday's event loop delay issues" and the skill automatically queries observability data, correlates CPU profiling, and generates an HTML report with data visualizations and root cause analysis. Transformed a process that required engineers to manually query multiple data sources into a one-sentence automated workflow.
  • Team adoption: Team members without backend backgrounds can now independently perform preliminary performance analysis through the skill, lowering the troubleshooting barrier.

Next Steps: Schedule the skill to run daily performance reports and automatically create tasks on the project management board, achieving end-to-end automation from "detect issue → analyze issue → assign task."

Value: This case demonstrates a complete loop — from solving a specific technical problem, to distilling troubleshooting patterns, to using AI to productize experience into reusable team capability. It's not simply "using AI to write code," but making AI an engineering operations lever.
